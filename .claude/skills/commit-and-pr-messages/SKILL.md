# Commit and PR messages

A lookup, not a workflow — consult the row that matches what you're drafting, then apply the rules below it.

| Drafting...           | What's different                                                                                                                                                                                                                                                                                                                                                                                           |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A commit message      | Header must be a valid Conventional Commit; never hand-append `(#NNNN)`; never hand-write a `chore(release):` message                                                                                                                                                                                                                                                                                      |
| A PR title            | Same rules as a commit message — this repo squash-merges, so the PR title _becomes_ the final commit header. **Unlike `maps-app`, this repo actually lint-checks the PR title in CI** (`.github/workflows/dhis2-verify-pr-title.yml`, via `JulienKode/pull-request-name-linter-action` against `@dhis2/cli-style`'s commitlint config) — a malformed title fails a real check, not just a style preference |
| A PR description body | No template exists here (`maps-app` has one, this repo doesn't) — write free-form prose matching the real pattern below                                                                                                                                                                                                                                                                                    |

## Conventional Commits format

`<type>(<scope>)?: <subject>` — enforced both locally (commitlint via `@dhis2/cli-style`) and in CI for the PR title specifically (see above).

-   **Allowed types**: `build`, `chore`, `ci`, `docs`, `feat`, `fix`, `perf`, `refactor`, `revert`, `style`, `test` — but real usage on `master` is dominated by `feat`, `fix`, and scoped `chore` variants (`chore(deps)`, `chore(deps-dev)`, `chore(release)`, `chore`). Don't reach for an exotic type just because it's in the allowed list if `feat`/`fix`/`chore` already fits.
-   Header ≤ 120 chars, no trailing period, not sentence-case/Start Case/PascalCase/UPPERCASE.

## Bot-generated patterns — recognize these, never hand-write them

-   `chore(release): cut X.Y.Z [skip ci]` — semantic-release only. **Note the exact suffix here is `[skip ci]`, not `[skip release]`** (that's `maps-app`'s convention, not this repo's — don't cross-contaminate).
-   `chore(deps): bump <pkg> from A to B (#NNNN)` / `chore(deps-dev): bump ...` / `chore(deps): bump the dependencies group with N updates` — Dependabot only.

## Jira ticket references

`[PROJECTKEY-NUMBER]` in square brackets, placed after the description, before any `(#PRNUM)`:

```
fix: skip temporal aggregation if collection is empty [DHIS2-20684] (#625)
feat: support method operations across bands and handle scale for small areas [DHIS2-20355] (#608)
feat: support temporal aggregation in Earth Engine worker [DHIS2-17856] [DHIS2-20149] (#607)
```

Multiple tickets in one bracket-stack is fine (last example above has two). A few older commits use `(DHIS2-XXXXX)` in parens instead — that's legacy, not the convention to follow; use square brackets.

## What makes a real "good" message here

Describes the observable behavior/outcome, not implementation mechanics:

```
fix: show labels for isolated points in cluster layers (#654)
fix: unmask eeImageBands [DHIS2-20276] (#606)
feat: azureLayer support [DHIS2-19152] (#600)
```

## Two things to never fabricate

-   **`(#PRNUM)`** — GitHub appends this automatically when the PR is squash-merged. Never write it into a drafted commit message or PR title yourself.
-   **`chore(release): cut X.Y.Z [skip ci]`** — generated by semantic-release on release. Never hand-write one.

## PR description body — real, free-form pattern (no template exists)

Real example (PR #654, `fix: show labels for isolated points in cluster layers`):

```markdown
## Description

#### Goal

Fix a bug where points in a clustered map layer (`Cluster` and its variants —
`ClientCluster`, `DonutCluster`, `ServerCluster`) lost their text label as soon as
they stood alone, i.e. weren't part of a cluster bubble and hadn't been spiderfied.

#### The bug

Before this fix, `Cluster.createLayers()` added a circle layer for non-clustered/leaf
points but never added any label layer for them...
```

Match this shape: `## Description`, then sub-headers as genuinely useful (`#### Goal`, `#### The bug`, `#### The fix`, etc.) — not a fixed checklist, not screenshots-by-default, not a Jira-link-first-line requirement (link the ticket inline in prose if relevant instead). Keep it concise and "what"-focused.
